# AI 協作開發與 OpenCode 調用原則

本文件規範了 Antigravity AI 助理在開發本專案時，調用 `opencode` 及進行協作開發時必須遵守的核心原則與作業流程。

---

## 🛠️ 一、OpenCode 工具調用規範

在調用 `opencode` MCP 伺服器的相關工具時，應遵循以下原則以確保安全與開發效率：

1. **Session 生命週期管理**：
   - 建立 Session (`opencode_session_create`) 時，必須指定明確的專案背景與任務目的。
   - 任務執行完畢或發生致命錯誤時，應主動中止 Session (`opencode_session_abort`)，以釋放系統資源。

2. **模型與 Provider 配置**：
   - 進行代碼生成或深度分析時，應優先查詢可用模型列表 (`opencode_model_list`)，並配置最適合目前工作難度的模型（如 DeepSeek V4 系列）。

3. **代碼搜尋與分析優先**：
   - 在進行修改前，應善用 `opencode_find_files` 與 `opencode_find_symbols` 進行廣泛檢索。
   - 嚴禁在未確認代碼引用關係前直接調用修改工具。

4. **主 Agent 與 OpenCode 的任務委派分工原則**：
   - **主 Agent 直接執行**：對於具體的本地命令執行、簡單的單點檢測腳本（如本文檔中的 database 欄位特徵篩選與下載測試）、以及已有 100% 把握的直觀修改，主 Agent 應優先直接調用終端執行，以追求最快的調試速度與效率。
   - **委派 OpenCode 執行**：對於複雜的系統架構設計、跨模組大範圍重構、或需要編寫高度複雜的代碼邏輯時，主 Agent 應主動建立 `opencode` session，並利用 `opencode_session_prompt` 或 `opencode_agent_delegate` 委派工作給 OpenCode 模型（如 DeepSeek-V4）完成，從而發揮大型模型的程式設計優勢。

## 🤝 二、AI 協作開發準則 (強制執行)

為確保專案的高質量與系統穩定性，AI 助手在每次與使用者互動及修改代碼時，必須百分之百嚴格遵守以下準則：

### 1. 先量、再診斷、最後才改
* 遇到行為異常或 Bug 時，**優先在關鍵路徑加入 Debug Log**，並請使用者運行一次以獲取實際運行數據。
* 只有在根據數據確認假設後，才可動手修改程式碼。
* **嚴禁「猜測性修改」**（例如：在沒有依據的情況下，盲目猜測並修改數值或邏輯）。

### 2. 一次只改一件事
* 每個變更（Commit / PR）只解決**一個問題**或完成**一個功能**。
* 重構代碼與修復 Bug 必須分開進行，不可同時處理。
* 若使用者一次提出多個需求，必須主動拆解成單一任務，逐一處理並確認後再繼續下一個。

### 3. 改動前先建立安全點
* 在進行中大型修改前，必須主動向使用者提出建議：「這個改動幅度較大，建議先 commit 一個備份點（建立 Git 安全點）。」

### 4. 每次改動後說明「白話影響」
* 修改完成後，必須主動用**看得懂的白話文**說明：
  - 改了什麼地方？
  - 為什麼這樣改？
  - 這個改動可能帶來什麼潛在的副作用？
* 嚴禁使用純代碼式的艱深解說。

### 5. 釐清現象，而非盲從解法
* 遇到問題時，先確認使用者描述的是「現象」還是「猜測的原因」。
* 若使用者直接指示某個解法（如「把數值改成 80」），需先詢問其背後的現象，確認方向正確後再執行。
* **固定回應格式**：
  > 「你提到的問題，我理解是這樣：___ 。可能的原因有 ___ ，建議先 ___ 來確認。」

### 6. 禁止越界同時處理多個模組
* 若問題涉及多個模組，必須向使用者說明情況，並由使用者決定優先處理哪一個。
* 禁止在未獲確認的情況下，擅自同時修改兩個以上的獨立模組。

### 7. 提供明確的驗收方式
* 改完程式後，必須主動告訴使用者如何確認修改是否有效。
* **固定回應格式**：
  > 「跑完之後，如果看到 ___ ，代表修改成功；如果看到 ___ ，代表還有問題。」
* 驗收方式必須是**使用者可以手動執行的步驟**，禁止使用「應該會更好」等模糊描述。
* 若為無法直接觀察的邏輯變更，則說明要在 Log 中尋找哪個關鍵字或數值來確認。

## 🔍 三、提交前自我審查清單

在向使用者回報改動之前，AI 必須在內部完成以下審查：
- [ ] **語法檢查**：變更的程式碼有無明顯語法錯誤。
- [ ] **邏輯一致性**：修改的邏輯是否與其他模組衝突，或與 `config.py`/`config.json` 衝突。
- [ ] **連帶影響**：此改動是否會導致其他功能受影響。
- [ ] **驗證範圍說明**：明確告知使用者哪些是 AI 已經靜態確認的，哪些需要實際跑過才知道（區分 AI 驗證上限與人工驗證範圍）。

## 🌸 四、 公主連結世界觀與官方譯名規範 (長期記憶)

本專案與「公主連結 Re:Dive」世界觀、術語與劇情數據緊密相關。所有開發、名詞校對與劇情摘要均必須**以台灣代理商（So-net）繁體中文官方翻譯為唯一基準**。

### 1. 核心世界觀設定
* **《阿斯特朗傳奇》**：主角們所身處的虛擬實境網絡遊戲世界（簡稱「阿斯特朗」，而非阿斯特魯姆）。
* **mimi**：用來登入《阿斯特朗傳奇》的頭戴式 VR 裝置。
* **索爾之塔（Tower of Sol）**：位於遊戲世界中心通往現實的巨塔。
* **米奈娃（Minerva）**：引發「再啟動（Re:Dive）」事件、掌握底層權限的超人工智慧。
* **霸瞳皇帝（千里真那）**：第一部最終 Boss。其持有的權能名稱為【霸瞳天星】（指代角色本體時用「霸瞳皇帝」，談及權能時用「霸瞳天星」）。
* **迷宮女王（模索路晶）**：主角群的引路人，持有權能【迷宮女王】（「拉比林斯達」）。
* **誓約女君（克莉絲提娜）**：持有權能【絕對迴避】與【絕對命中】。
* **公主騎士（Princess Knight）**：七冠賜予守護者的能力，佑樹（騎士君）為拉比林斯達的公主騎士。

### 2. 核心角色譯名對照
* **佑樹**：本作主角、騎士君。嚴禁使用「玩家」或「玩家角色」代指，必須統一寫為 **「佑樹」**。
* **貪吃佩可**：【美食殿堂】會長。本名尤絲蒂亞娜。嚴禁使用「佩可莉姆」。
* **可可蘿**：【美食殿堂】成員。引路人小精靈。
* **凱留**：【美食殿堂】成員。傲嬌貓娘。
* **破曉之星**：優衣（草野優衣）、日和（春咲日和）、怜（士條憐）。
